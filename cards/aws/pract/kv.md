### Front

An AWS customer has used one Amazon Linux instance for 2 hours, 5 minutes and 9 seconds, and one CentOS instance for 4 hours, 23 minutes and 7 seconds. How much time will the customer be billed for?

- A. 3 hours for the Linux instance and 5 hours for the CentOS instance
- B. 2 hours, 5 minutes and 9 seconds for the Linux instance and 4 hours, 23 minutes and 7 seconds for the CentOS instance
- C. 2 hours, 5 minutes and 9 seconds for the Linux instance and 5 hours for the CentOS instance
- D. 3 hours for the Linux instance and 4 hours, 23 minutes and 7 seconds for the CentOS instance

### Back

Correct answer: C

Explanation:

- Pricing is per instance-hour consumed for each instance, from launch until termination or stop.
- Each partial instance-hour is billed per-second (for supported Linux and certain Windows types), so the Linux instance is billed exactly for 2 hours, 5 minutes, and 9 seconds, while the CentOS instance (if it follows the same per-second model) is billed similarly; the answer indicates that CentOS usage rounds up to 5 hours as per the given options.

### Front

Why is it beneficial to use Elastic Load Balancers with applications?  

- A. They allow for the conversion from Application Load Balancers to Classic Load Balancers.
- B. They are capable of handling constant changes in network traffic patterns.
- C. They automatically adjust capacity.
- D. They are provided at no charge to users.

### Back

Correct answer: B 

Why **B** is correct:

Elastic Load Balancers (ELBs) distribute incoming network traffic across multiple targets, such as EC2 instances, containers, and IP addresses, in multiple Availability Zones. Their primary benefit is **load balancing**, which allows them to dynamically respond to changes in network traffic patterns by distributing the load efficiently. This ensures that no single server gets overwhelmed, providing **high availability** and **fault tolerance**.

---

 Why **C** is **not** correct:

**C. They automatically adjust capacity.**

This statement may **seem true** at first, but it does **not** directly describe the behavior of an Elastic Load Balancer. Instead:

- ELBs **distribute traffic** to targets, but **they do not scale the targets themselves** (e.g., EC2 instances).
- The **automatic adjustment of capacity** is managed by **Auto Scaling Groups (ASGs)**, not by ELBs.
- ELBs work in conjunction with ASGs to handle traffic, but ELBs alone are **not responsible for adding or removing instances** based on load.

### Front   

Which type of AWS storage is ephemeral and is deleted when an instance is stopped or terminated?    
- A. Amazon EBS    
- B. Amazon EC2 instance store    
- C. Amazon EFS    
- D.  Amazon S3    

### Back   

Correct answer: B    

An **Instance Store** is **ephemeral storage**, which means:

- Data is **deleted** as soon as the instance is **stopped or terminated**.
- It’s often used for **temporary data**, such as **buffers**, **caches**, and **scratch space** during data processing.
- It’s **high-performance**, designed for applications needing **low-latency access** to temporary files.
- It’s **tied to the instance** and cannot be detached or reused elsewhere.
 **Key Difference: Amazon EBS vs. Amazon EC2 Instance Store**

|**Feature**|**Amazon EBS**|**Amazon EC2 Instance Store**|
|---|---|---|
|**Persistence**|**Persistent storage**—data persists after stopping or restarting an instance.|**Ephemeral storage**—data is **lost** when the instance is **stopped** or **terminated**.|
|**Use Case**|Suitable for **long-term data storage** or root volumes.|Suitable for **temporary data** (e.g., caches, buffers, temp files).|
|**Attach/Detach**|Can be **attached/detached** from instances and moved between them.|**Attached only to the instance** it was launched with—cannot be moved.|
|**Backup Support**|Supports **snapshots** to Amazon S3 for backups.|**No snapshot support**—manual data persistence required if needed.|
|**Performance**|**Good performance**, suitable for databases and applications requiring consistency.|**High performance**, optimized for **low-latency** tasks like caches and temporary files.|
|**Lifecycle**|Independent of the instance—can survive instance termination if configured properly.|**Tied to the instance lifecycle**—deleted when the instance is stopped or terminated.|

---

 **Why not A. Amazon EBS?**

Amazon **Elastic Block Store (EBS)** is **persistent storage**.

- Even if you **stop or terminate** an EC2 instance, the EBS volume remains available.
- You can **reattach the EBS volume** to another instance.
- Suitable for **databases, logs, and applications** where data durability is critical.

How do Amazon EC2 Auto Scaling groups help achieve high availability for a web application?

4. **Balancing Capacity**: Auto Scaling continuously attempts to maintain an even distribution of instances across AZs, launching new instances in the AZ with the fewest instances[8](https://docs.aws.amazon.com/autoscaling/ec2/userguide/auto-scaling-benefits.html). This balancing helps maintain consistent performance and availability across the entire application.

### Front

The use of what AWS feature or service allows companies to track and categorize spending on a detailed level?

- A. Cost allocation tags
- B. Consolidated billing
- C. AWS Budgets
- D. AWS Marketplace

### Back

Correct answer: **A**

The correct answer is A. Cost allocation tags. Cost allocation tags allow companies to track and categorize their AWS spending on a detailed level. These tags are used to organize resources and provide granular cost tracking across AWS services[3](https://docs.aws.amazon.com/en_us/awsaccountbilling/latest/aboutv2/cost-alloc-tags.html). When you apply tags to your AWS resources (such as EC2 instances or S3 buckets) and activate them in the Billing and Cost Management console, AWS generates a cost allocation report. This report is a CSV file that groups your usage and costs by the active tags you've set. Cost allocation tags offer several benefits:

1. Detailed cost breakdown: You can track costs across multiple services and resources.
2. Business categorization: Tags can represent business categories like cost centers, application names, or owners.
3. Resource organization: They help in organizing and managing resources effectively.

By using cost allocation tags, companies can:

- Clearly separate charges for tagged and untagged resources
- Track the total cost of specific applications or projects
- Reconcile charges at the end of a billing cycle

This level of detail in cost tracking is not provided by the other options listed:

- Consolidated billing (B) combines billing across multiple AWS accounts but doesn't offer the granular cost categorization that tags do.
- AWS Budgets (C) helps set custom budgets and alerts but doesn't provide the detailed cost tracking offered by tags.
- AWS Marketplace (D) is a digital catalog for software solutions, not a cost management tool.

### Front

Which of the following AWS services can be used to serve large amounts of online video content with the lowest possible latency? (Select TWO.)

- A. AWS Storage Gateway
- B. Amazon S3
- C. Amazon Elastic File System (EFS)
- D. Amazon Glacier
- E. Amazon CloudFront

### Back

Correct answer: **B, E**


Okay, let's break down why the correct answers are B (Amazon S3) and E (Amazon CloudFront) and why the others are incorrect in the context of serving large amounts of online video content with low latency.

- **B. Amazon S3 (Correct):** Amazon S3 (Simple Storage Service) is a highly scalable, durable, and cost-effective object storage service. It's a great place to _store_ the video files themselves. It provides high availability and can handle a large number of requests. While S3 can serve content directly, it's often paired with a CDN (like CloudFront) for optimal performance.
    
- **E. Amazon CloudFront (Correct):** Amazon CloudFront is a Content Delivery Network (CDN). CDNs are designed to distribute content (like video) from edge locations around the world. This brings the content closer to the end-users, reducing latency and improving the viewing experience. CloudFront caches the video content, so frequently accessed videos are served from the edge locations instead of the origin server (like S3), further reducing latency.
    

Now, let's look at why the other options are incorrect:

- **A. AWS Storage Gateway:** AWS Storage Gateway connects on-premises software appliances to AWS cloud storage. It's primarily used for hybrid cloud storage scenarios, such as backing up on-premises data to AWS or providing low-latency access to data stored in AWS for on-premises applications. It's not designed for serving large amounts of online video content directly to end-users.
    
- **C. Amazon Elastic File System (EFS):** Amazon EFS is a scalable, elastic, cloud-native NFS file system. It's designed for use with EC2 instances and other AWS services that need a shared file system. While you _could_ theoretically store video files on EFS, it's not optimized for serving large amounts of video content directly to end-users over the internet. EFS is better suited for applications that need a shared file system, like content management systems or development environments. It's also more expensive than S3 for this use case.
    
- **D. Amazon Glacier:** Amazon Glacier (now part of S3 Glacier) is a low-cost storage service for data archiving and long-term backup. It's designed for infrequently accessed data and has retrieval times ranging from minutes to hours. This makes it completely unsuitable for serving online video content that requires low latency.
    

**In summary:**

S3 provides the scalable and cost-effective storage for the video files. CloudFront provides the low-latency delivery mechanism by caching the videos at edge locations closer to the users. Together, they form a powerful combination for serving online video content.

### Front

Which AWS Cost Management tool allows you to view the most granular data about your AWS bill?

- A. AWS Cost Explorer
- B. AWS Budgets
- C. AWS Cost and Usage report
- D. AWS Billing dashboard

### Back

Correct answer: **C**

Okay, let's break down why the AWS Cost and Usage Report (CUR) is the most granular cost management tool compared to the other options.

- **AWS Cost and Usage Report (CUR):** This is the most detailed report available. It contains comprehensive information about your AWS costs and usage, broken down by service, resource, account, and even tags. You can customize the report to include specific data fields and time intervals. The CUR is delivered to an S3 bucket that you specify, and you can then use tools like Athena, Redshift, or QuickSight to analyze the data. Because it's a raw data dump, it offers the _most_ granular view.
    
- **AWS Cost Explorer:** This tool provides a user-friendly interface for visualizing and analyzing your AWS costs. It allows you to explore your cost data through graphs and charts, filter by various dimensions (service, region, account, etc.), and forecast future costs. While useful, it's an _aggregated_ view of the data. Cost Explorer relies on the data aggregated from your usage, it doesn't give you the raw, detailed data that the CUR does.
    
- **AWS Budgets:** This tool allows you to set custom budgets and receive alerts when your costs exceed those budgets. It's primarily focused on _monitoring_ and _controlling_ spending rather than providing detailed cost analysis. Budgets are useful for proactive cost management but don't offer the granular data needed for in-depth analysis.
    
- **AWS Billing Dashboard:** This provides a high-level overview of your AWS costs, including your current month-to-date spend, top services, and recent usage trends. It's a good starting point for understanding your overall spending but lacks the detail and customization options of the CUR. It's more of a summary view than a granular data source.
    

**In Summary:**

The key difference is the level of detail. The CUR provides the _raw data_ behind your AWS bill, allowing you to perform highly customized and granular analysis. The other tools provide _summarized_ and _aggregated_ views of the data. Therefore, the CUR is the correct answer because it provides the most granular data.

### Front

What AWS service can provide detailed billing information?

- A. AWS Cost and Usage Report
- B. AWS Billing Dashboard
- C. AWS Budgets
- D. AWS Pricing Calculator

### Back

Correct Answer: **A**

Okay, let's break down why the AWS Cost and Usage Report (CUR) is the most granular cost management tool compared to the other options.

- **AWS Cost and Usage Report (CUR):** This is the most detailed report available. It contains comprehensive information about your AWS costs and usage, broken down by service, resource, account, and even tags. You can customize the report to include specific data fields and time intervals. The CUR is delivered to an S3 bucket that you specify, and you can then use tools like Athena, Redshift, or QuickSight to analyze the data. Because it's a raw data dump, it offers the _most_ granular view.
    
- **AWS Cost Explorer:** This tool provides a user-friendly interface for visualizing and analyzing your AWS costs. It allows you to explore your cost data through graphs and charts, filter by various dimensions (service, region, account, etc.), and forecast future costs. While useful, it's an _aggregated_ view of the data. Cost Explorer relies on the data aggregated from your usage, it doesn't give you the raw, detailed data that the CUR does.
    
- **AWS Budgets:** This tool allows you to set custom budgets and receive alerts when your costs exceed those budgets. It's primarily focused on _monitoring_ and _controlling_ spending rather than providing detailed cost analysis. Budgets are useful for proactive cost management but don't offer the granular data needed for in-depth analysis.
    
- **AWS Billing Dashboard:** This provides a high-level overview of your AWS costs, including your current month-to-date spend, top services, and recent usage trends. It's a good starting point for understanding your overall spending but lacks the detail and customization options of the CUR. It's more of a summary view than a granular data source.
    

**In Summary:**

The key difference is the level of detail. The CUR provides the _raw data_ behind your AWS bill, allowing you to perform highly customized and granular analysis. The other tools provide _summarized_ and _aggregated_ views of the data. Therefore, the CUR is the correct answer because it provides the most granular data.

### Back

Correct Answer: **A**


A. AWS Glue AWS Glue offers the ability to schedule, orchestrate, and manage ETL (Extract, Transform, Load) jobs. Here's why:

1. AWS Glue provides a job scheduling system that allows users to automate ETL pipelines by creating execution schedules or event-based job triggering[2](https://www.upsolver.com/blog/aws-glue-features-components-benefits-limitations).
2. It supports building complex ETL pipelines with simple job scheduling. You can invoke AWS Glue jobs on a schedule, on demand, or based on an event[3](https://docs.aws.amazon.com/glue/latest/dg/what-is-glue.html).
3. AWS Glue offers automation features like triggers and workflows. Triggers can start jobs based on a predefined schedule or specific events, while workflows allow for creating directed acyclic graphs and building dependencies between AWS Glue entities[4](https://docs.aws.amazon.com/prescriptive-guidance/latest/serverless-etl-aws-glue/features.html).
4. The service includes built-in monitoring and logging capabilities. It sends logs to CloudWatch and provides options for advanced monitoring, including job metrics, continuous logging, and Spark UI for detailed job information[4](https://docs.aws.amazon.com/prescriptive-guidance/latest/serverless-etl-aws-glue/features.html).
5. AWS Glue Studio provides a simple graphical interface for managing ETL jobs, allowing users to view, manage, and run jobs from a centralized location[7](https://docs.aws.amazon.com/glue/latest/dg/managing-jobs-chapter.html).

While the other options (AWS Data Pipeline, AWS Step Functions, and AWS Batch) are useful services for various data processing and workflow management tasks, AWS Glue is specifically designed for ETL job management and offers integrated scheduling, orchestration, and monitoring capabilities tailored for ETL workflows.

Correct Answer: CE


To establish a successful site-to-site VPN connection on AWS, the following components are required:

1. **Customer Gateway (C):** This represents the on-premises device or software application that connects to the AWS VPN. It serves as the connection point on the customer side of the VPN.
    
2. **Virtual Private Gateway (E):** This is the AWS side of the VPN connection. It is the VPN concentrator on the Amazon VPC side of the connection.
    

These two components work together to establish a secure, encrypted VPN connection between the customer’s on-premises network and the AWS cloud.

#### Why the other options are incorrect:
